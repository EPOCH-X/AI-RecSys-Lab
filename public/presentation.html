Problem

무엇을 풀고 싶은가

User journey

사용자가 겪는 흐름

홈→ 질문 (무드·활동·장르·템포·에너지·자유 서술)→ 로딩→ 클라이언트: runRecommendationGraph→ POST /api/catalog/songs→ 결과 Top-5→ 알고리즘 비교 (발표용)

NormalizedUserProfile로 정규화 → (선택) Gemini로 서술 기반 태그 보강 → 프로필로 검색 쿼리 여러 개 생성 → iTunes로 곡 수집 → 곡마다 Last.fm 태그(가능 시) → 내부 Song으로 변환 후 점수화.

스택: Next.js(App Router) · Zustand · TypeScript. 추천 그래프: src/graph/recommendationGraph.ts · 공통 파이프라인: recommendationPipeline.ts

Data

어디서 데이터를 가져오나

Apple iTunes Search API 제목·아티스트·앨범·아트·미리듣기 등. 프로필에서 최대 6개 검색어(장르/무드/활동/템포/에너지 조합)로 여러 번 검색 후 중복 제거 병합.
Last.fm (선택, LASTFM_API_KEY) 곡별 태그. 실패 시 빈 배열 — 전체는 죽지 않음. 태그가 비면 무드/BPM/에너지 추정이 비슷해져 점수 동률이 늘 수 있음.
로컬 JSON users.json 가상 사용자, playlists.json 큐레이션. 곡 ID가 전부 숫자(iTunes)면 협업 부스트 스킵 — 로컬 데모 ID와 섞이지 않게 하기 위함.
Google Gemini NEXT_PUBLIC_GEMINI_API_KEY 등. 프로필 보강 + 추천 이유 문장. 순위 결정 알고리즘은 아님.

Pipeline

순위를 만드는 블록들

블록 역할
콘텐츠 (scoreSongs) 장르·무드·활동 태그 매칭. 규칙 기반이라 발표 시 설명 용이.
협업 부스트 가상 사용자 중 유사 이웃의 liked/skipped 반영. iTunes 숫자 ID 카탈로그에서는 미적용.
플레이리스트 보정 큐레이션과 맞는 후보 가산.
컨텍스트 BPM 선호 구간, 운동/휴식, 에너지 일치, 로맨틱 태그 등.
가중 합성 각 점수 축을 max로 정규화 후 0.5·0.3·0.2 (콘텐츠·협업·컨텍스트).
Cosine 프로필·곡 희소 벡터 코사인 → finalScore 대체.
MMR 상위 풀(예: 25곡)에서 λ·관련성·상호 유사도로 Top-5 선택.

Content score

콘텐츠 점수 (Baseline의 첫 단계)

구현: src/domain/scoring/index.ts — 태그는 부분 문자열 매칭도 허용.

  • 장르: 프로필 장르와 곡 장르가 맞으면 +3
  • 무드: 곡 moodTags 중 프로필 moods와 겹치는 개수 × 2
  • 활동: 곡 activityTags 중 프로필 activities와 겹치는 개수 × 2
  • contentScore = 위 합. 같은 조합이면 동점이 잦음 → 컨텍스트·가중합·(또는) Cosine/MMR이 차별에 기여.

Baseline merge

최종 점수 합성 (모드: baseline / hybrid_mmr 전 단계)

nc = content / max(content)  ·  nCol = max(collab,0) / max(collab⁺)  ·  nCtx = max(context,0) / max(context⁺)
blended = 0.5×nc + 0.3×nCol + 0.2×nCtx  →  finalScore = blended × 10

  • max 정규화 때문에 상위권이 비슷하게 보일 수 있음 — 발표에서 “트레이드오프”로 언급 가능.
  • hybrid_mmr는 위 finalScore까지 동일하게 계산한 뒤, Top-5만 MMR로 다시 고름.

Algorithm modes

실행 모드 4종

NEXT_PUBLIC_RECSYS_ALGO = baseline | cosine | cosine_mmr | hybrid_mmr  ·  변경 후 dev 서버 재시작

Baseline (A) 콘텐츠 → 협업(조건부) → 플레이리스트 → 컨텍스트 → 가중합. 기본 데모·설명용.
Cosine (B) 콘텐츠 점수 단계 후 코사인만으로 재랭킹. “유사도만 쓰면?” 대조군.
Cosine + MMR (B+D) 코사인 점수 상위 풀 → MMR로 Top-5. 다양성 강조 실험.
Hybrid + MMR (A+D) Baseline과 동일 점수 후 MMR만 적용. 정확도 논리 유지 + 목록 다양화.

Comparison

알고리즘 비교 — 발표용 한 장

모드 강점 약점 / 주의 언제 쓰나
Baseline 해석 쉬움, 블록별 기여 설명 가능 정수 콘텐츠·정규화로 동점·근접 빈번 기본 시연, “왜 이 곡?” 슬라이드
Cosine 수식 명확, ML 입문 대비 특성이 비슷하면 유사도도 뭉침 Baseline과의 정량·정성 비교
+ MMR 비슷한 곡 연속 완화, 다양성 지표 설명 용이 λ·풀 크기에 따라 순위 변동 “다양성 vs 관련성” 트레이드오프

Cosine & MMR

짧은 정의

코사인 유사도 프로필과 곡을 같은 축(장르·무드·활동·BPM·에너지 등)의 벡터로 두고, 각도에 가까운 개념인 cos(θ)로 “얼마나 같은 방향인가”를 0~1에 가깝게 측정.
MMR 매 선택마다 “지금 후보의 점수(관련성)”와 “이미 고른 곡들과의 유사도”를 λ로 섞어, 상위만 골랐을 때 생기는 편중을 줄임.

코드: alternativeAlgorithms.ts — applyCosineRanking, applyMmrRerank

LLM

Gemini 역할 (순위와 분리)

순위에 가까운 영향 프로필 보강 → 태그·검색 품질에 간접 영향.
순위와 무관 이미 정해진 Top-N에 대한 자연어 이유 생성.
  • 발표 문장 예: “추천 순서는 규칙·유사도·MMR이 정하고, Gemini는 사용자에게 보여 줄 설명을 씁니다.”

비교 API: 동일 입력으로 4모드 결과만 빠르게 보려면 LLM 이유를 생략할 수 있음.

Evaluation

평가 지표 아이디어 (발표용)

  • 다양성: Top-K에서 서로 다른 장르/아티스트 수 (앱 비교 화면과 동일 개념).
  • 커버리지: 여러 세션에서 추천된 곡이 카탈로그의 몇 %를 덮는지.
  • 정성: 팀원 설문 “맞는 것 같다” 척도 — 같은 입력으로 모드별 A/B.
  • 지연: 카탈로그 API 응답 시간( iTunes + Last.fm 병렬량에 민감).

오프라인 정답 라벨이 없으면 NDCG 등은 생략하고, 위 항목만으로도 학부 수준 발표는 충분히 가능.

Limitations

한계 · 개선 로드맵

  • API 변동: 타임아웃·재시도·캐시 없음 → 느리거나 간헐 실패.
  • Last.fm 공백: 태그 폴백, iTunes primaryGenreName 활용, 호출 동시 수 제한.
  • 동점: 콘텐츠에 BPM·에너지 거리 가산, tie-break 규칙.
  • 보안: 클라이언트에 노출되는 API 키는 반드시 도메인/쿼터 제한.

Closing

정리

질문으로 만든 프로필과 실시간 카탈로그 위에서 Baseline · Cosine · MMR 조합 4모드를 돌릴 수 있고, 발표용 화면으로 동일 입력 비교와 마크다운보내기까지 지원합니다. LLM은 순위가 아니라 보강·설명에 집중합니다.
